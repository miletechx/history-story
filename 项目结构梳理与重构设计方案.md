# 历史故事会 - 项目结构重构方案（精简版）

## 1. 项目概要

一个 Next.js 16 全栈应用，核心功能是"用户输入历史主题 → AI 生成故事 → TTS 语音播放 → 记录和排行"。

技术栈：Next.js 16 App Router / React 19 / TypeScript 5 / Tailwind CSS 4 / shadcn/ui / Supabase / coze-coding-dev-sdk / pnpm

核心功能链路：

```
首页(选年龄+输入主题) → 播放页(查缓存→AI生成→TTS→播放→保存记录) → 排行榜
```

## 2. 当前结构和问题

### 2.1 现有目录

```
src/
├── app/          # 页面路由 + API 路由
├── components/   # shadcn/ui + 业务组件混放
├── hooks/        # 自定义 hooks
├── lib/          # 工具 + 常量 + 类型 + 配置 + 模拟数据 全部混放
└── storage/      # 数据库 schema + Supabase client
```

### 2.2 核心问题

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| 页面太胖 | 首页 500+ 行，播放页 600+ 行，UI/状态/业务/数据全写一起 | 改一处容易影响其他 |
| API 无分层 | route.ts 里同时做参数校验、业务逻辑、数据库操作 | 逻辑重复、难复用 |
| 常量文件混杂 | constants.ts 放了类型、配置、模拟数据、正则 | 找东西靠搜索 |
| 数据类型不一致 | schema 里 duration 是 text，前端传 number，排行榜用 parseInt 解析 | 潜在 bug |
| SEO 未适配 | 根布局还是"扣子编程"模板内容 | 不匹配产品 |

## 3. 重构原则

1. 不改功能、不改路由地址、不改数据库表结构、不改 API 对外地址
2. 页面只负责"摆组件"，业务逻辑进 hooks 和 services
3. 分层要简单：不搞 service + repository 两层，合并为一层 services（含数据库操作）
4. 前端请求封装和服务端业务逻辑命名要能区分开
5. 迁移分阶段进行，每阶段可独立验证

## 5. 关键设计决策

### 5.1 为什么不要 repository 层？

当前项目体量小，数据库操作简单（基本是单表 CRUD），service 直接调用 Supabase 就够了。

```
API route → service（含数据库操作）→ 返回
```

不需要：

```
API route → service → repository → 返回
```

等某个模块数据访问真的复杂了（比如多表联查、事务），再单独拆，不提前过度设计。

### 5.2 前端请求封装和服务端 service 如何区分？

| 位置 | 用途 | 命名 |
|------|------|------|
| `features/auth/api-client.ts` | 前端调用 API 的 fetch 封装 | `api-client.ts` |
| `services/auth.service.ts` | 服务端业务逻辑 + 数据库 | `xxx.service.ts` |

规则：
- `api-client.ts` = 浏览器端运行，调用 `/api/xxx`
- `xxx.service.ts` = Node.js 端运行，被 API route 调用

### 5.3 service 函数必须接收 request headers

这是本项目的特殊约束。TTS 和 AI 生成依赖 `coze-coding-dev-sdk`，需要从请求中提取转发头：

```ts
// services/story-generation.service.ts
import { HeaderUtils } from 'coze-coding-dev-sdk';

export function generateStory(
  query: string,
  ageGroup: AgeGroup,
  requestHeaders: Headers  // ← 必须从 route.ts 传入
) {
  const customHeaders = HeaderUtils.extractForwardHeaders(requestHeaders);
  // ...
}
```

```ts
// app/api/generate-story/route.ts
import { generateStory } from '@/services/story-generation.service';

export async function POST(request: NextRequest) {
  const { query, ageGroup } = await request.json();
  return generateStory(query, ageGroup, request.headers);
}
```

迁移时如果忘了传 headers，AI 生成和 TTS 会直接失效。这是最容易踩的坑。

### 5.4 顺带修正的数据一致性问题

| 问题 | 现状 | 建议 |
|------|------|------|
| duration 类型混乱 | schema 定义 text，前端传 number，排行榜 parseInt | 统一为 integer 类型 |
| story_records.user_id | 用 serial() 定义，实际是外键 | 改为 integer() |

这些可以在第四阶段（拆 service）时一起修正，配合数据库迁移脚本。

## 6. 各层职责一句话说明

| 层 | 职责 | 一句话 |
|----|------|--------|
| app/ | 路由入口 | 像门牌号，不写业务 |
| features/ | 前端业务模块 | 看文件夹名就知道是什么功能 |
| services/ | 服务端业务 | AI 生成、TTS、数据库操作都在这里 |
| config/ | 配置 | 年龄段、音色、SEO |
| types/ | 类型定义 | 全项目复用 |
| components/ui/ | 基础 UI | shadcn 不动 |
| components/common/ | 通用业务组件 | Header、Toast |
| hooks/ | 通用 hooks | 跨功能复用 |
| lib/ | 工具函数 | cn()、formatTime() |
| storage/ | 数据库层 | schema + client |

## 7. 分阶段迁移计划

### 第一阶段：拆配置和类型（风险：低）

```
src/lib/constants.ts
  ↓ 拆成：
src/types/age.ts          → AgeGroup 类型
src/types/story.ts        → Story, StoryContent 类型
src/config/age-groups.ts  → AGE_GROUPS, AGE_GROUP_CONFIG
src/config/tts-voices.ts  → TTS_VOICES
src/lib/story-id.ts       → generateStoryId()
src/lib/age-group.ts      → normalizeAgeGroup()
```

原 constants.ts 改为 re-export，保持现有引用不报错：

```ts
// src/lib/constants.ts（过渡期保留）
export * from '@/types/age';
export * from '@/types/story';
export * from '@/config/age-groups';
export * from '@/config/tts-voices';
```

验证方式：项目能正常 build，所有页面功能不受影响。

### 第二阶段：拆首页（风险：中低）

```
src/app/page.tsx（500+行）
  ↓ 拆成：
src/app/page.tsx                              → 3 行入口
src/features/home/home-page.tsx               → 首页主逻辑
src/features/home/components/HeroSection.tsx   → Hero 区域
src/features/home/components/AgeSelector.tsx   → 年龄选择
src/features/home/components/StorySearchBox.tsx → 搜索框
src/features/home/components/HomeRankingSection.tsx → 排行榜区
src/features/home/data/ranking-mock.ts         → 模拟排行榜数据
src/features/home/data/search-suggestions.ts   → 推荐搜索项
```

拆完后 app/page.tsx 只剩：

```tsx
import { HomePage } from '@/features/home/home-page';
export default function Page() {
  return <HomePage />;
}
```

验证方式：首页外观和功能完全不变。

### 第三阶段：拆播放页（风险：中）

```
src/app/play/page.tsx（600+行）
  ↓ 拆成：
src/app/play/page.tsx                              → 3 行入口
src/features/player/player-page.tsx                → 播放页主组件
src/features/player/hooks/useStoryGeneration.ts    → 故事生成流程
src/features/player/hooks/useAudioPlayer.ts        → 播放控制
src/features/player/hooks/usePlayRecord.ts         → 播放记录保存
src/features/player/hooks/useStoryHistory.ts       → 本地历史
src/features/player/components/AudioPlayer.tsx     → 播放器 UI
src/features/player/components/StoryContentCard.tsx → 故事正文
src/features/player/components/GenerationProgress.tsx → 生成进度
```

验证方式：播放页能正常生成故事、播放音频、保存记录。

### 第四阶段：拆 API service（风险：中）

```
每个 API route.ts
  ↓ 抽出业务逻辑到：
src/services/xxx.service.ts
```

重点注意：

- service 函数签名必须接收 `requestHeaders: Headers`
- 数据库操作直接在 service 里写，不需要额外 repository

示例：

```
src/app/api/tts/route.ts
  变成：接参数 → 调用 tts.service.ts → 返回响应

src/services/tts.service.ts
  负责：选音色 → 调 TTSClient → 返回音频 URL
```

验证方式：所有 API 接口返回结果不变。

### 第五阶段：收尾优化（风险：低）

- 根布局 metadata 改为"历史故事会"
- html lang 改为 `zh-CN`
- 抽取统一 AppHeader 组件
- 修正 duration 字段类型（配合数据库迁移）
- 排行榜接口改用数据库聚合查询

## 8. 迁移注意事项

### 绝对不能忘的事

1. **headers 传递**：所有调用 coze SDK 的 service 必须接收原始 request.headers
2. **re-export 过渡**：拆 constants.ts 时保留原文件做 re-export，避免大面积改引用
3. **'use client' 标记**：features 里的组件如果用了 useState/useEffect，必须标记
4. **每阶段独立验证**：每拆完一个阶段就跑一遍完整功能，确认无误再继续

### 绝对不要做的事

- 不要一次性移动所有文件
- 不要同时改结构和改功能
- 不要改 API URL 路径
- 不要改数据库表名
- 不要引入新的状态管理库
- 不要重写播放器逻辑

## 9. 重构后的阅读路径

小白按这个顺序看项目，5 分钟就能理解整体：

```
1. src/app/page.tsx              → "首页入口在这里"
2. src/features/home/            → "首页长什么样、有什么功能"
3. src/features/player/          → "播放页怎么工作的"
4. src/features/player/hooks/    → "故事怎么生成、音频怎么播放"
5. src/services/                 → "服务端怎么调 AI、怎么存数据"
6. src/config/                   → "项目有哪些配置"
7. src/types/                    → "项目有哪些数据类型"
8. src/storage/database/         → "数据库表长什么样"
```

## 10. 改造优先级汇总

| 优先级 | 内容 | 风险 | 收益 |
|--------|------|------|------|
| P0 | 拆 types + config | 低 | 消除混杂，全项目复用 |
| P1 | 拆首页 | 中低 | 首页可读性提升 |
| P2 | 拆播放页 | 中 | 最复杂模块变清晰 |
| P3 | 拆 API → service | 中 | 后端可维护性提升 |
| P4 | SEO + 体验收尾 | 低 | 产品完整度提升 |

## 11. 总结

这次重构的核心思想：

```
不是重写，是整理。
不加新功能，只让现有功能住进更合理的"房间"。
```

整理完之后：
- 每个文件职责单一，改起来不怕牵连
- 看目录名就知道功能在哪
- 新增功能只需要在对应 feature 里加文件
- 服务端逻辑集中，排查问题快
